The scene graph system in OpenGeometry stores named collections of geometric entities inside the
kernel. In public applications, `OGSceneManager` is most useful when you want to:

- keep a long-lived scene snapshot in wasm
- project a multi-entity model to 2D
- export that model to STL, STEP, or IFC

Unlike a Three.js scene, `OGSceneManager` stores **B-Rep snapshots**, not live mesh references.

## Overview

The scene graph consists of two primary structures defined in `main/opengeometry/src/scenegraph.rs`:

- **OGScene**: a named collection of serialized entities
- **OGSceneManager**: the public wasm-facing manager for scenes, entities, projection, and export

## OGScene

A scene is a named container for entities:

```rust
pub struct OGScene {
    pub id: String,
    pub name: String,
    pub entities: Vec<SceneEntity>,
}
```

### SceneEntity

Each entity stores a B-Rep payload plus application-facing identifiers:

```rust
pub struct SceneEntity {
    pub id: String,
    pub kind: String,
    pub brep: Brep,
}
```

The `kind` field is an app-facing label. It can match the wrapper type (`"Cuboid"`, `"Solid"`,
`"Polygon"`) but it does not have to.

## Public workflow

For Three.js applications, the public workflow is wrapper-first:

1. Create wrapper geometry such as `Cuboid`, `Solid`, `Opening`, or `Polygon`.
2. Serialize its B-Rep payload.
3. Add that payload to `OGSceneManager`.
4. Project or export the stored scene.

```ts
import { Cuboid, OGSceneManager, OpenGeometry, Vector3 } from "opengeometry";

await OpenGeometry.create({ wasmURL: "/opengeometry_bg.wasm" });

function toBrepSerialized(source: unknown) {
  const value =
    source && typeof (source as { getBrepSerialized?: () => string }).getBrepSerialized === "function"
      ? (source as { getBrepSerialized: () => string }).getBrepSerialized()
      : source && typeof (source as { getBrepData?: () => unknown }).getBrepData === "function"
        ? (source as { getBrepData: () => unknown }).getBrepData()
        : source && typeof (source as { getBrep?: () => unknown }).getBrep === "function"
          ? (source as { getBrep: () => unknown }).getBrep()
          : source;

  return typeof value === "string" ? value : JSON.stringify(value);
}

const manager = new OGSceneManager();
const sceneId = manager.createScene("Building");

const cuboid = new Cuboid({
  center: new Vector3(0, 1.5, 0),
  width: 5,
  height: 3,
  depth: 2,
  color: 0x10b981,
});

manager.addBrepEntityToScene(sceneId, "box-1", "Cuboid", toBrepSerialized(cuboid));
```

## Scene lifecycle

### createScene()

```ts
const sceneId = manager.createScene("Floor Plan");
```

### setCurrentScene() / getCurrentSceneId()

```ts
manager.setCurrentScene(sceneId);
const currentSceneId = manager.getCurrentSceneId();
```

### listScenes()

Returns a JSON array of scene summaries.

```ts
const summaries = JSON.parse(manager.listScenes());
```

### removeScene()

```ts
const removed = manager.removeScene(sceneId);
```

## Entity management

### addBrepEntityToScene() / addBrepEntityToCurrentScene()

These are the most general public entity entrypoints:

```ts
manager.addBrepEntityToScene(sceneId, "wall-1", "Cuboid", toBrepSerialized(cuboid));
manager.addBrepEntityToCurrentScene("wall-2", "Cuboid", toBrepSerialized(cuboid));
```

### replace / refresh helpers

When you want to overwrite an existing entity ID, these helpers are also available:

- `replaceBrepEntityInScene(...)`
- `replaceBrepEntityInCurrentScene(...)`
- `refreshBrepEntityInScene(...)`
- `refreshBrepEntityInCurrentScene(...)`

They all keep the same public idea: update a stored scene entity using a serialized B-Rep payload.

### Primitive-specific add helpers

`OGSceneManager` also exposes typed add helpers such as:

- `addLineToScene(...)`
- `addPolylineToScene(...)`
- `addArcToScene(...)`
- `addRectangleToScene(...)`
- `addPolygonToScene(...)`
- `addCuboidToScene(...)`
- `addCylinderToScene(...)`
- `addSphereToScene(...)`
- `addWedgeToScene(...)`

These are handy when you are already working with lower-level kernel objects. For wrapper-first
apps, `addBrepEntityToScene(...)` is usually the most consistent approach.

## Projection and export

Scenes can be projected to 2D with hidden line removal:

<Warning>
2D projection and hidden line removal are experimental. Output and options may change.
</Warning>

```typescript
const cameraJson = JSON.stringify({
  position: { x: 8, y: 6, z: 8 },
  target: { x: 0, y: 0, z: 0 },
  up: { x: 0, y: 1, z: 0 },
  near: 0.1,
  projection_mode: "Orthographic",
});

const hlrJson = JSON.stringify({
  hide_hidden_edges: true,
});

const scene2dJson = manager.projectTo2DCamera(sceneId, cameraJson, hlrJson);
const scene2d = JSON.parse(scene2dJson);
```

The same stored-scene model is used for export:

```ts
const stl = manager.exportSceneToStl(sceneId);
const step = manager.exportSceneToStep(sceneId);
const ifc = manager.exportSceneToIfc(sceneId);
```

## Serialization and snapshots

Scenes are serializable snapshots:

```ts
const sceneJson = manager.getSceneSerialized(sceneId);
const removed = manager.removeEntityFromScene(sceneId, "box-1");
```

This makes `OGSceneManager` useful for export pipelines, collaboration snapshots, and derived 2D
documents, but it also means scene entities are not live references back to your Three.js meshes.
